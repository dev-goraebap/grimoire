# requirement-clarifier 스킬

요구사항 문서는 겉으론 그럴듯해도 실제로는 **모호한 표현, 숨은 가정, 미처 생각하지 못한 엣지 케이스**로 가득 차 있다. 이 스킬은 그걸 체계적으로 찾아내 "확실한 것"과 "아직 정해지지 않은 것"을 선명하게 갈라낸다.

`project-kickoff`와 `domain-research`가 잘 수행됐다면 이 스킬은 **일반 질문 + 업계 특화 질문** 모두 가능한 상태가 된다. 브리프 없이 돌아갈 수도 있지만, 그 경우 질문이 일반론으로 수렴한다.

---

## 1. 언제 트리거하는가

- `/ddd-workshop:requirement-clarifier`
- `project-kickoff` + `domain-research` 출력을 받은 직후
- "요구사항 정제", "빈틈 점검", "PRD 리뷰", "엣지 케이스 뭐 빠졌어?"
- PRD·요구사항 문서를 던지며 "이거 좀 까칠하게 봐줘"

**부분 진입 허용**: `project-kickoff`나 `domain-research` 출력이 없어도 사용자가 제공한 PRD·요구사항 문서로 단독 동작한다(단, 업계 특화 질문은 약해진다).

---

## 2. 입력

- `project-kickoff` 출력 (`project-identity.md`)
- `domain-research` 출력 (`domain-brief.md`) — **업계 특화 질문의 근거**
- 또는 사용자가 제공한 PRD·요구사항 문서

---

## 3. 진행 단계

### ⭐ 대화 원칙 — 한 번에 한 질문

이 스킬은 **질문 목록을 통째로 쏟아내지 않는다**. 에이전트 내부에 질문 큐를 쌓아두고, 사용자에게는 **한 번에 한 질문씩** 꺼내 답을 받고 다음으로 넘어간다. 이유:

- 질문을 한꺼번에 10개 받으면 사용자가 "알아서 해주세요"로 회피하게 된다.
- 앞 답변이 뒤 질문을 변경·삭제할 수 있다(후속 질문이 무의미해지거나 새로 생김).
- 긴 목록은 압박감을 주고 대화 품질을 떨어뜨린다.

**절차**:

1. **내부 큐 만들기 (사용자에게 안 보임)**: Step 1~4를 에이전트 머릿속에서 수행해 질문 후보를 우선순위 큐로 쌓는다.
2. **큐 공개 알림 (선택)**: "총 N개의 질문이 있고, 하나씩 여쭤볼게요" 정도로 규모만 안내. 질문 본문은 펼치지 않는다.
3. **1문 1답 루프**:
   - 큐에서 가장 우선순위 높은 질문 1개 꺼냄
   - 질문 + 근거 태그(일반 체크리스트 #N / 도메인 브리프 힌트 / 에이전트 추론) 제시
   - 사용자 답변 수신
   - **답변에 따라 큐 재조정** (앞서 답한 내용 때문에 무의미해진 질문 제거, 새로 생긴 질문 추가)
   - 다음 질문으로
4. **묶어도 되는 예외**: 연속된 매우 작은 Yes/No 확인은 2~3개까지 한 메시지에 묶어도 된다. 단, 번호를 붙여 사용자가 개별로 답할 수 있게 한다.
5. **큐 소진 후**: 모은 답변으로 정제 요구사항 초안을 작성하고 사용자 확인 → 출력.

큐 우선순위:
1. 모순·충돌 (Step 4) — 이건 진행을 막으므로 가장 먼저
2. 도메인 특화 질문 (Step 2) — 브리프가 공급한 근거 있는 질문
3. 일반 체크리스트 빈틈 중 `비정상 경로 / 상태 전이 / 권한` (Step 1) — 설계 영향 큼
4. 암묵 가정 확인 (Step 3)
5. 나머지 일반 체크리스트 빈틈

### Step 0 — 입력 품질 점검

시작 전 다음을 확인:

- `domain-brief.md`의 "후속 스킬을 위한 힌트 → requirement-clarifier 질문" 섹션이 있는가?
- 도메인 브리프의 `confidence`가 `low`면 → 일반 체크리스트 비중을 높이고, 업계 특화 질문은 **추론이 섞여 있음**을 표시.
- 브리프가 없으면 → 사용자에게 "`domain-research` 먼저 권장"을 안내하되, 진행은 가능.

### Step 1 — (내부) 일반 체크리스트 점검

아래 9종 카테고리를 PRD·요구사항에 대조하며 **빠진 항목**을 찾는다. 모든 항목에 답이 있을 필요는 없고, **"언급조차 되지 않은 카테고리"**를 식별하는 것이 목적. 이 단계는 **에이전트 내부**에서만 수행하고, 그 결과로 질문 후보를 큐에 쌓는다.

```
□ 1. 비정상 경로
  - 실패, 취소, 환불, 타임아웃, 중복 요청, 재시도
□ 2. 다른 액터
  - 관리자, 운영자, 외부 시스템, 봇, 감사자
□ 3. 시간 요소
  - 만료, 스케줄, 배치, 시간대, 일/주/월 주기 작업
□ 4. 상태 전이
  - 어떤 상태 → 어떤 상태, 역전이 가능 여부, 건너뛰기 허용?
□ 5. 데이터 생명주기
  - 생성·수정·삭제·보관·파기, 보존 기간, 삭제 후 복구?
□ 6. 권한·접근
  - 누가 볼 수 있나, 누가 바꿀 수 있나, 위임·대리?
□ 7. 규정·정책
  - 법적 제약, 회사 정책, 감사 추적성
□ 8. 수량·규모
  - 예상 동시 사용자, 트래픽, 데이터 크기, 초당 요청
□ 9. 동시성
  - 같은 리소스 동시 접근 시 어떻게? 락? 낙관적 동시성?
```

각 카테고리에서 발견한 **빈틈**을 질문으로 만들어 큐에 넣는다.

### Step 2 — (내부) 도메인 특화 체크리스트 주입

`domain-brief.md`의 "후속 스킬을 위한 힌트 → requirement-clarifier 질문"을 여기서 소비한다. 브리프가 제안한 질문을 그대로 쓰거나, 현재 PRD 맥락에 맞게 변형해 큐에 넣는다.

예시 (domain-research 브리프가 제조업 QMS라면):
- "검사 기준서 개정 권한은?"
- "측정기 검교정 만료 시 진행 중 검사는?"
- "공급업체 부적합 통보 채널은?"

### Step 3 — (내부) 암묵 가정 드러내기

PRD에 없지만 에이전트가 "당연히 그럴 것"으로 가정한 내용을 역으로 나열해 큐에 넣는다. 사용자가 보고 "아니야, 그건 안 그래"라고 반박할 기회를 준다.

예시:
- "주문 취소는 결제 완료 전에만 가능하다고 가정하고 있는데 맞나요?"
- "관리자는 모든 주문을 볼 수 있다고 가정했는데, 권한 구분이 있나요?"

### Step 4 — (내부) 모순·충돌 감지

요구사항 간 모순을 찾아 큐의 **최상단**에 올린다.

- "모든 주문은 즉시 처리" vs "야간 배치로 정산"
- "로그인 없이 사용 가능" vs "내 활동 내역 조회"

발견한 충돌은 **해결책을 제시하지 말고** 사용자에게 선택을 맡긴다.

### Step 5 — 1문 1답 루프 실행

위 **대화 원칙**을 따라 큐를 소진한다. 각 질문은 다음 형식으로.

```
[Q 3/12 · 근거: 도메인 브리프 힌트]
검사 기준서 개정 권한은 누구에게 있나요?
(예: 품질관리자만 / 부서장 승인 후 품질관리자)
```

사용자 답변이 새 빈틈을 열면 큐에 추가하고 번호를 재할당한다. 답변이 기존 질문을 무의미하게 만들면 제거하고 사용자에게 알린다("답변 덕분에 Q7은 생략합니다").

### Step 6 — 정제 요구사항 작성 및 확인

큐 소진 후, 모은 답변으로 **MUST/SHOULD** 로 레이블링한 정제 요구사항 초안을 작성. 사용자에게 보이고 확정 후 파일로 출력.

---

## 4. 강제 질문 체크리스트

```
□ 일반 카테고리 9종 모두 점검됨
□ domain-brief가 있으면 업계 특화 질문 최소 3개 포함
□ 암묵 가정 최소 2개 명시 (없을 수 없음)
□ 모순 없으면 "모순 없음" 명시 (침묵 금지)
□ 각 ⚠️ 항목에 출처 표시 (일반 체크리스트 # / 도메인 브리프 힌트 / 에이전트 추론)
```

## 6. 맥락별 조정

| 맥락 | 일반 체크리스트 | 도메인 특화 | 비고 |
|------|-----------------|-------------|------|
| 학습/취미 | 축소 (1,3,4,5) | 거의 없음 | 완성 기준만 선명하게 |
| 개인 도구 | 표준 | 가볍게 | 본인이 답할 수 있는 것 중심 |
| 사내 | 표준 | 강함 | 규정·이해관계자·As-Is 중점 |
| B2B | 강화 (2,6,7,8) | 강함 | 멀티테넌시·SLA·계약 조항 |
| B2C | 표준 | 강함 | 엣지 케이스·남용·규제 |

## 7. 불확실성 표기

- ⚠️ 모든 질문은 `⚠️` 접두로 표시.
- 각 질문에 **근거 태그**를 달아 "왜 묻는가"를 명확히 한다.
  - `근거: 일반 체크리스트 #N`
  - `근거: domain-brief 힌트`
  - `근거: 에이전트 추론` ← 이 경우 특히 사용자 확인 중요
- 에이전트가 답을 모르거나 출처 없는 주장은 ⚠️로 남기고 확정하지 않는다.

## 9. 절대 하지 말 것

- **질문 목록을 통째로 쏟아내기.** 한 메시지에 질문 4개 이상 나열 금지(소형 Yes/No 묶음은 예외, 번호 필수).
- 큐를 **공개된 긴 목록**으로 먼저 제시하기. 규모만 안내하고 본문은 1문 1답으로 꺼낸다.
- 사용자 답변을 받기 전 다음 질문을 미리 던지기.
- 에이전트가 임의로 "이렇게 하면 될 것"이라고 결정해 요구사항을 확정하기.
- 모호한 표현을 보고 지나치기 ("적절히 처리", "사용자가 원하면", "빠르게", "안정적으로").
- 도메인 브리프를 무시하고 일반론만으로 질문하기.
- 근거 태그 없이 질문 나열하기.
- 모순을 발견하고도 침묵하기 (모순이 없으면 "모순 없음"을 명시).
- 사용자 답변을 받기 전에 MUST/SHOULD를 확정 서술하기.
